03·PROBLEM What agentic curation actually demands

The thesis

Agentic curation needs structured memory,
not Markdown sprawl.

— WHAT MOST AGENT HARNESSES DO

Save context to MD files or a RAG index. Adequate for chat, but less powerful when you need to ask structured questions (How many? How much?) or require tracking complex context at scale.

— WHAT SKILLFUL ALHAZEN DOES

Typed entities, typed relations, typed notes — persistent across sessions, queryable across skills.

— the lesson, recalled

Bioinformatics taught us data hygiene 20 years ago. This was the justification for biomedical ontologies previously — the same issues apply now.

07·SKILLS The unit of extension

Anatomy of a skill

A skill is a self‑contained directory — and a marketplace entry.

Anatomy

skills/jobhunt/
├── SKILL.md            // triggers · workflows · commands
├── skill.yaml          // metadata · namespace prefix
├── schema.tql          // TypeDB extension
├── jobhunt.py          // the commands that Claude calls
├── quality-checks.yaml
└── dashboard/          // optional Next.js

Single source of truth: skills-registry.yaml. Core skills with path:, externals with git:. make build-skills resolves the registry, drops symlinks, loads schemas.

Contract

Every skill contributes four things:

①A typed schema namespace — every type prefixed. ②A Python CLI — scripts + functions the agent invokes. ③Agent‑readable SKILL.md — triggers, workflows, commands. ④Optional dashboard — Next.js components wired at build.

Marketplace pattern

Skills live in their own git repos - requiring simple registration into local config.

08·HARD PROBLEM · 1 Ontological distinctions create clarity
01

Hard problem · one

Use ontological distinctions to create clarity in the data.

An LLM left to its own linguistic instincts will say: “an author is‑a person, so make Author a subclass of Person.” That reads fine in English — and it falls over the moment a real schema has to hold it.

Naive — LLM default

Author sub Person

Reviewer, Patient, Employee all want the same slot. TypeDB is single‑inheritance — Alice can’t be all four at once. Attributes leak onto Person. An ex‑author was never really a person.

Sophisticated — BFO / UFO

(Alice : Person) bears (Author : Role)

Person stays rigid & intrinsic. Role is anti‑rigid & externally grounded — it can start, end, and coexist with other roles without touching Alice’s identity.

Rigidity · the test that exposes the difference

DISPOSITION Internally grounded — fragility, metabolic tendency. Survives even if the social world vanishes.
ROLE Externally grounded — job‑seeker, author, defendant. Strip the institution — the role evaporates, the person remains.

Ontological engineering buys downstream clarity — for the schema, for queries, and for the agent reasoning over it. Surrendering to “standard LLM linguistic logic” quietly bakes in errors you only find at scale.

Implementation

</> alhazen_notebook.tql 
// the rigid object — identity-bearing
entity person
   · given-name
   · family-name
   · email

// the anti-rigid hook — externally grounded
entity role
   · status          // active / ended
   · started-on
   · ended-on

// the inherence relation — how a role attaches
relation role-bearing
   · bearer          → person
   · borne-role      → role

// skills extend the leaves — never the trunk
jhunt-job-seeker   sub   role
scilit-author      sub   role
apm-patient        sub   role
① Role data lives on the role, never on the person. ② Skills extend alh-role — jhunt-job-seeker, scilit-author. ③ Lifecycle attrs capture when a role was realized.

09·HARD PROBLEM · 2 Schema as a living artifact
02

Hard problem · two

The schema can't be frozen — the agent learns new things.

Standard agent stacks treat "I can't represent this" as a silent skip. We treat it as the most important signal in the system.

Two failure modes

EXECUTION Tool crash · empty result · API down
→ standard error log + retry
SCHEMA GAP TypeDB write rejected · concept has no type
→ filed as a structured GitHub issue

The evolution loop

[1] Claude writes to TypeDB → error code [SYR1] [TYR01] [FEX1]
[2] PostToolUse hook detects the code, prompts [SCHEMA-GAP-HINT]
[3] file-schema-gap CLI: dedupes, routes to repo, opens issue
[4] gap-triage.yml labels by severity / phase / skill
[5] Local fix · validate against running TypeDB · draft PR · human merge

Schema gaps are signal, not noise. They are how the graph grows from use, not from upfront design.

10·HARD PROBLEM · 3 A lightweight runtime, on purpose
03

Hard problem · three

Don't build an agent platform. Use the one you already pay for.

— constraints

  • ·Existing Claude license — keep token cost low
  • ·One engineer — maximum flexibility, minimum maintenance
  • ·Deploy must be git clone && make build
  • ·No platform lock‑in

— moves

  • ·Claude Code is the agent runtime
  • ·Deploy is git clone && make build
  • ·One runtime, many skills

Tradeoff accepted

Less control over the agent loop. We don't own the orchestrator — a Claude Code change can move things under us.

Win banked

~10× less code to maintain. Onboarding a new skill is one PR. Iteration speed is the moat.

11·HARD PROBLEM · 4 Notebook memory over Markdown sprawl
04

Hard problem · four

Prove the notebook is actually better than what people already do.

Markdown notes

Drift & duplicate.

Same fact in three files, slightly different. No way to ask “which”.

RAG only

Opaque blob.

Useful for retrieval, useless for audit. Can't introspect what's in there.

TypeDB notebook

Typed & queryable.

Multi‑hop questions across skills. Provenance on every claim.

Example query — only possible because the schema is real

// "Every contact who flagged a skill gap, grouped by funding stage."
match  $c isa jhunt-contact,        has name $cname;
       $g isa jhunt-skill-gap-note,  has content $gtxt;
       $co isa jhunt-company,        has stage $stage;
       (mentioner: $c, mention: $g) isa jhunt-mention;
       (employee: $c, employer: $co) isa alh-works-at;
fetch  $cname; $stage; $gtxt;

RAG & MD are still here when they're the right tool — the notebook adds structure on top, not instead.
